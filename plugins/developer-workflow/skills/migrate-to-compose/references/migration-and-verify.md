# Migration Constraints & Verification — Detailed Reference

## Phase 6: Migration Constraints

Key constraints to enforce on the `compose-ui-architect` agent:

### Business Logic Preservation — Critical

- **Do not touch** repositories, use cases, domain models, data layer, or any class not directly involved in rendering or user interaction
- **ViewModel changes allowed only when necessary**: adapting the state type, adding an `onAction` dispatcher if one didn't exist, or switching a legacy stream type (e.g. `LiveData` → `StateFlow`) if agreed with the user
- If you want to refactor something outside the UI layer — don't. Note it as a separate suggestion.

### Architecture

- Stateless screen composable: `FooScreen(state: FooState, onAction: (FooAction) -> Unit)`
- `viewModel()` called only at the navigation entry point
- No business logic inside composables
- State classes **must** have `@Stable` or `@Immutable` if the project uses explicit stability annotations
- `when` expressions over sealed types must be **exhaustive — no `else` branch**
- Error/message strings in state must use the same type as the rest of the project's state classes
- Parameterless actions: follow the project convention; if none found, prefer plain `class` or `data class` with no fields

### Code Quality

- Inline composable lambdas longer than ~8 lines must be extracted to private composable functions
- Composable function bodies over ~50 non-empty lines should be split into sub-composables
- Visibility: `internal` by default; `private` for helpers; `public` only for cross-module entry points

### Theming and Resources

- Sizes and dimensions: use project theme tokens or named constants, never raw `dp` literals
- String resources: if the project uses `stringResource()`, all strings must go through resources
- Colors: `MaterialTheme.colorScheme.X` or project tokens — never raw hex values

### Previews and Comments

- Every significant composable gets a `@Preview`
- Multiple `@Preview` functions for distinct states
- Comments for non-obvious decisions only — *why*, not *what*

### Screenshot Tests (if agreed in Phase 4)

One test per visual state. Each renders the stateless composable in isolation with hardcoded `FooState(...)`. Use the project's existing tool (Paparazzi or Roborazzi). Record initial snapshots as baseline.

## Phase 7: Static Verification

### Step 1: Build, lint, and screenshot tests

- Compile: `./gradlew :<module>:compileDebugKotlin` — zero errors
- Lint: `./gradlew :<module>:lintDebug` — no new errors or warnings
- Screenshot tests: `recordPaparazziDebug` / `recordRoborazziDebug` to record, then `verify*` on subsequent runs

### Step 2: Migration fidelity review

Go through `behavior-scenarios.md` and the original XML element by element:

**Layout structure and spacing**
- Every View in XML has a corresponding Compose element
- Margins/paddings reflected as `Modifier.padding` — none silently dropped
- Element order preserved
- `ConstraintLayout` relationships correctly expressed

**Visual properties**
- Text styles match: `textAppearance`, `textSize`, `textColor` → correct theme tokens
- Backgrounds, shapes, corner radii, elevation transferred
- Drawables/icons mapped correctly
- Tint colors preserved

**Behavioral details easy to miss**
- Fragment-level setup in `onViewCreated` (toolbar, menus, shared elements)
- Activity-level setup in `onCreate` (window flags, result contracts, intent extras, edge-to-edge)
- `tools:` attributes that hint at runtime behavior
- `<include>` or `<merge>` layouts fully represented
- Accessibility: `contentDescription`, `importantForAccessibility`, `labelFor` preserved

**Architecture**
- Screen composable is stateless
- `when` over sealed types is exhaustive
- Old Activity/Fragment/View kept intact

**Window insets**
- Edge-to-edge: `enableEdgeToEdge()` if original used it
- Text input: `Modifier.imePadding()` on scrollable container
- `fitsSystemWindows="true"` → `Scaffold` or `systemBarsPadding()`
- No hardcoded system bar heights

## Device Testing

Brief the `manual-tester` agent with:
- The `behavior-scenarios.md` from Phase 1
- The migration report's visual comparison table
- The list of interactions to verify

The agent captures before/after screenshots, executes all test cases, compares layout/typography/colors/spacing, and populates the screenshot table in the report.

## Post-migration Cleanup

Once verification passes, **offer to clean up** (do not do automatically):

Files to propose deleting:
- Old Activity / Fragment class
- XML layout file(s)
- RecyclerView adapters fully replaced
- ViewHolder classes
- DataBinding / ViewBinding generated classes
- DiffUtil.Callback implementations
- `attrs.xml` entries for migrated custom Views

Also review:
- Navigation graph: remove Fragment destination, add Compose equivalent
- DI / Hilt modules: remove Fragment-scoped bindings

**Run project-wide usage search before proposing deletion of any class or resource.**
